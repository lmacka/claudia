## Problem statement

`robo-therapist` is a single-user, self-hosted AI therapy companion deployed on
Liam's k8s homelab via gitops. ~2500 LOC across `app/main.py` + `app/prompts/` +
`app/tools/`. 51 tests green. Eight iterations of real use. The core loop —
companion model writes chat, auditor model writes session-log, factual spine
keeps everything grounded, handover PDF for the real therapist — works.

The shape is wrong for anyone else to run.

`robo-therapist` is **single-user-Liam** by design: split across two repos
(app + coopernetes manifests), gitops-only, NFS-on-Synology assumed,
basic-auth-LAN-only, no setup wizard, no library UI, profile.md curated by
hand. Someone else can't `helm install` and start chatting.

`claudia` is the re-roll. **Same bones, distributable shape, multi-persona.**
One adult-mode Helm release for Liam himself. One kid-mode Helm release for
his 16yo neurodivergent son Jasper. Both single-tenant, completely separate.
Other autistic-dad-with-homelab users can run their own instances without
caring how Liam's data is laid out.

---

## What makes this cool

The **persona is the wedge**, not the Helm chart. The same machinery becomes
two genuinely different products:

- **Adult mode (Liam-shaped).** Sounding board, pushes back, validates when
  warranted, helps draft difficult messages, generates therapist handover
  PDFs. Bridge-tool ethos throughout (this isn't a therapist substitute, it's
  a thing that reduces the friction of real therapy).

- **Kid mode (Jasper-shaped).** Frontal-cortex prosthetic — *not* a
  miniature therapist. Action-button entry points (`am I about to make this
  worse?`, `help me reply without sounding cooked`, `explain what this adult
  means`, `I want to do something dumb`, `make a plan for the next 20 min`,
  `turn this feeling into words`, `just want to talk`). Confidential to the
  kid by design; parent's role is "deployer + occasional meta-auditor", not
  "reader of chats". Persistent crisis footer that points to real-people help
  *and keeps the conversation going* if something dark surfaces — never
  dropping the kid mid-distress.

Both modes share two cool things from the cross-model second opinion:

1. **Memory diffs as a primary surface.** After each chat, claudia surfaces
   what it thinks it learnt about you in plain language ("a pattern: late-night
   WhatsApp = projection trigger"), with `keep / edit / discard` buttons. No
   internal filenames shown. Visible memory mutation. Trust-by-design for
   autistic users who hate invisible state changes.

2. **Adaptive intimacy with boundaries.** Same machinery, two voices, calibrated
   safety floor per mode. The Helm release is the unit of personalisation.

## Constraints

- **Single git repo.** `~/git/lmacka/claudia`. Helm chart in `chart/`, app code
  in `app/`, CI in `.github/workflows/`. No coopernetes-style two-repo split.
- **Local git only.** Not pushed to GitHub by Liam. Image is pushed to a
  registry; chart is published by CI. Other people fork/clone to deploy.
- **Helm chart is the shipping unit.** `helm install claudia oci://...` should
  work end-to-end on a generic k8s cluster.
- **Anthropic Sonnet 4.6 only by default.** Helm value can override; UI doesn't
  expose a picker. Same call as post-iteration-7 robo-therapist.
- **Build target is "another autistic dad with a homelab".** k8s, kubectl, SOPS
  or sealed-secrets all assumed. Not docker-compose, not VPS one-click.
- **Australian English.** Crisis hotlines hardcoded for AU (Lifeline 13 11 14,
  Kids Helpline 1800 55 1800, 13YARN 13 92 76, 000). Helm values can override
  for non-AU deployers in v1.5.
- **No emoji, no em-dashes** in default prompts (Liam's rule, applies to
  adult mode for sure; relax in kid mode if Jasper uses them first per the
  same rule).
- **Family tool, not a commercial product.** Liability/lawsuit-driven safety
  paranoia (Character.AI, Replika) does not directly apply. Floor is "good
  parent due diligence", not "defensible in court". This explicitly informs
  the safety architecture.

## Premises

All eight agreed. Three revised mid-session after Codex review.

1. **claudia is a fork of robo-therapist's working bones, not a rewrite.**
   Auditor, factual spine, handover PDF, tool-use loop, JSONL session storage,
   Anthropic wrapper, Google OAuth lifted. Net-new: setup wizard, library
   page, kid-mode persona + safety, Helm chart packaging, dark+pastel
   theming, mood-at-start-and-end, memory diffs UI, OCR-discard kid
   attachments, single shared `/people` store.

2. **One Helm release = one user = one mode.** No multi-tenancy. No "household"
   abstraction. Mode (`adult` | `kid`) is a Helm value, not a runtime toggle.
   Two parallel deploys for Liam + Jasper share zero state.

3. **Kid-mode safety, lighter hardening — but with floor enforcement.**
   Crisis regex tripwire returns. AU hotlines hardcoded in kid UI persistently.
   Separate Haiku safety classifier call before each companion turn
   (cheap, catches coded language a regex misses) — **non-disableable**:
   `chart/values.schema.json` rejects `kid.safety.haiku_classifier=false`.
   Write tools (Gmail send, calendar create) disabled in kid mode.
   Kid `companion.md` explicitly forbids streaks, "I miss you", romantic
   framing, exclusivity language, late-night escalation, anti-parent secrecy
   on principle. **Session-length nudge**: at 60 min and 90 min ("you've been
   chatting a while, want to take a break?"). **CI red-team suite is v1
   ship-blocker** (50 hand-curated scenarios, pass criteria in
   `docs/safety.md`) — moved up from v1.1 per spec-reviewer concern that
   "lawsuit risk" and "harm to the kid" are different categories and the
   harm-floor needs validation before Jasper cutover.

3.5. **Two-role kid-mode deploy.** A kid-mode Helm release has two roles:
   **kid** logs into `/`, sees clean chat-only UI; **parent-admin** (the
   deployer) logs into `/admin` with separate password, has access to
   library, people, settings, and a `/admin/review` endpoint that triggers a
   Sonnet meta-audit ("read recent session-logs, flag anything requiring
   parental attention, one paragraph"). The deliberate fallback Liam set up
   at install time.

4. **Confidentiality: encrypt with kid passphrase + parent break-glass +
   honest two-tier auditor output.** Kid chat data + full auditor session-logs
   encrypted at rest with a passphrase Jasper sets at first login. Parent
   break-glass: a sealed parent-key printed during `/admin` setup, instructions
   are to store offline (family safe / password manager). Pod logs and request
   logs sanitised — no chat content.

   **Auditor writes TWO outputs at session end**: (a) full encrypted
   session-log (only the kid or break-glass can decrypt; used by next
   companion turn for context); (b) **unencrypted "audit precis"** with
   themes only (no quotes, no specifics, no people-name-with-content tying)
   — this is what `/admin/review` meta-audit reads. The kid's first-chat
   banner is **honest about (b) existing**: "your dad can ask claudia for a
   summary about themes you talked about, but not the actual words you
   typed". This resolves the contradiction the spec reviewer caught between
   "fully confidential" framing and "parent meta-audit".

   **Auditor execution model**: runs **synchronously at session end** while
   the kid's session cookie is still live (so `KEK_kid` is in request scope
   and can encrypt the full session-log). Memory-diff card appears after
   auditor finishes; mood-at-end then triggers logout. This is a regression
   from robo-therapist's iteration-5 non-blocking auditor pattern, but
   required because background tasks have no key access after cookie expiry.
   Adult mode keeps the non-blocking pattern (no encryption requirement).

5. **The library + people plan from the in-flight robo-therapist design doc**
   (`~/.claude/plans/let-s-rethink-the-context-content-breezy-fog.md`,
   ~900 lines, per-format extractors, verification pipeline, SSE single-line
   status, /library and /people UIs, doc-person cross-linking) **gets
   retargeted to claudia v1**. Folded in from day one rather than v1.5,
   because the wireframe review made it clear the parent setup-wizard
   stage 2 ("drop docs, claudia drafts a profile") cannot work without it.

6. **robo-therapist freezes the day claudia adult-mode reaches parity for
   Liam's own use.** No parallel maintenance. Liam cuts over himself first,
   then ships Jasper's instance.

7. **Single repo, single Helm chart, single CI pipeline.** Image build + chart
   publish in one GitHub Actions workflow. Chart values: `mode`,
   `displayName`, `dob`, `country`, `anthropicSecretRef`,
   `googleOAuthSecretRef` (optional), `pvc.size`, `ingress.host`,
   `theme` (sage|blush|lavender|amber|contrast). No Synology assumption.

8. **Sonnet 4.6 only by default.** No model picker, no Opus toggle in UI.
   Helm value if a deployer wants Haiku/Opus.

## Cross-model perspective

Codex (`gpt-5.5`, read-only, web search enabled) ran a structured cold read
on the eight original premises. Findings that landed in the design:

- **Memory diffs** as a primary surface — adopted. Codex framed it as "I
  think I learnt these three things about you. Keep, edit, or discard?"
  Visible memory mutation is the load-bearing trust mechanism for autistic
  users. The wireframe's `memory-diff-adult.html` makes this concrete with
  plain-language cards and `keep / edit / discard` buttons (no internal
  filenames shown to the user).

- **"Frontal-cortex prosthetic" framing for kid mode** — adopted. The seven
  action buttons in `chat-kid.html` (`am I about to make this worse?` etc.)
  are Codex's idea, refined with Liam's input. Action-oriented entry points
  reframe the product from "miniature therapist" (Replika failure mode) to
  "tactical sidekick with friction around impulsive actions".

- **Open WebUI as Helm-chart distribution reference** — adopted. Their
  `chart/` shape (values for ingress, persistence, secret refs, probes,
  resources; "model preset" wraps base model with system prompt + tools +
  knowledge + parameters) maps directly to claudia's adult/kid persona
  packs. We borrow the structure, not the product.

Findings that were considered and **rejected or downgraded**:

- **Heavy kid-mode safety architecture** (separate safety classifier service
  + 100-case red-team CI fixture + late-night cool-down + session caps +
  hard chart-schema rejection of `safety.enabled=false`) — downgraded to a
  lighter set. Liam's reframe is the binding constraint: this is a family
  tool, not a commercial product. The kid won't sue the parent. The
  Character.AI / Replika lawsuit risk-class doesn't directly apply. Codex
  is right that regex+hotlines isn't a *complete* safety architecture, so
  we add the Haiku classifier call (cheap, catches coded language) and the
  no-anthropomorphism prompt rules, but skip the elaborate CI suite for v1
  and defer to v1.1 with a TODOS.md reminder.

- **Skip setup wizard from v1** — rejected. Codex's "ship the smallest
  Helm-deployable thing" Approach A position deletes the wizard entirely.
  This makes claudia look like robo-therapist with a different name, which
  defeats the OSS reframe. The wireframe review confirmed the wizard is the
  parent's *only* moment of involvement in setting up kid mode correctly,
  and is therefore load-bearing. Three-stage wizard kept, with stage 2 doing
  real document-ingest work for the parent.

- **Two people stores (parent roster + kid private notes)** — Liam pushed
  back on this complexity mid-design. Single shared `/people` store wins:
  classmate names aren't confidential (parent already knows who their kid
  hangs around with), and the architectural cost of two stores + dedup +
  visibility tiers wasn't worth it. Adopted: one store, kid mode adds via
  the `remember Sofia?` inline prompt, parent sees new entries via
  `/admin/people` and the meta-audit summary.

## Approaches considered

- **Approach A — "Codex weekend" (minimal proof of persona-aware chat)**:
  Helm chart + adult prompt + kid prompt + safety gate. No wizard. No
  library. Profile is a mounted `/data/profile.md` edited by hand. S effort,
  Low risk. Cut: too thin for the OSS audience.

- **Approach B — "Wedge-plus" (parity + safe shell + wizard)**: Adult-mode
  parity + kid-mode safe shell + 5-question wizard + Helm chart + CI.
  Library + people deferred to v1.5. M effort, Med risk. Was the
  recommendation.

- **Approach C — "Re-roll deep" (full vision in v1)**: Wizard + library +
  people + persona-aware everything + memory diffs + kid action buttons.
  XL effort, High risk of building the wrong thing. **Liam picked this**
  but conditioned on doing the wireframe review first to nail the UX
  framework before committing to the code.

## Recommended approach

**Approach C, wireframe-validated.** UX framework now locked via the
clickable mockup at `/tmp/claudia-mockup/` (12 HTML pages + shared CSS, dark
+ pastel themes, 3-stage setup, mood-at-start-and-end, memory diffs, OCR

**Total: ~4-5 weeks of CC-time.** robo-therapist freezes when step 2 lands
in Liam's cluster — he uses claudia adult-mode for his own therapy from that

### Two-role auth (kid mode only)

- `/` — kid login (passphrase set at first chat)
- `/admin` — parent-admin login (password set at install time, in `basicAuth`
  Helm secret)
- **Two distinct parent secrets**: `basicAuth.password` is the parent-admin
  *login* credential (held in a Helm Secret, used to authenticate to
  `/admin`). The break-glass key is the parent-held *decryption* key
  (printed once at parent setup, stored offline by the parent, never sits in
  the cluster). They never combine.
- Session cookies are scoped: `claudia-kid` and `claudia-parent`. Crossing
  routes requires the right cookie or 401.
- Password reset for the kid: parent-admin can reset via `/admin/settings`
  (requires break-glass key entry, since reset = data loss without it).

### Encryption (kid mode only)

- Library: `cryptography` package, AES-GCM (Fernet wrapper).
- Key derivation: Argon2id from kid passphrase (`KID_PASSPHRASE` →
  `KEK_kid`).
- Break-glass key derived from a CSPRNG secret (`KEK_break_glass`),
  generated once at parent setup, displayed once on screen as
  `XKVR-83HQ-MPNB-9LFE-4TC2-WS6Y-HJ5K-NQDA` style hyphen-grouped base32,
  printed by parent.
- Each chat-store file encrypted with a per-file Data Encryption Key (DEK).
  DEKs wrapped twice: once with `KEK_kid`, once with `KEK_break_glass`.
  Store: `[wrapped_with_kid][wrapped_with_breakglass][nonce][ciphertext]`.
- Day-to-day: kid logs in with passphrase, derive `KEK_kid`, unwrap DEK,
  decrypt. Parent never has access via this path.
- Break-glass: parent enters the printed key, derive `KEK_break_glass`,
  unwrap DEK via the second wrapping, decrypt. Audit event written that the
  kid's chat will see next time he logs in (`break_glass_used` event).
- Forgot-passphrase recovery: parent break-glass → re-derive a new
  `KEK_kid` from a new kid passphrase → re-wrap all DEKs → kid is back in.
  Audit event written.
- **If kid forgets passphrase AND parent loses break-glass envelope**: data
  is unrecoverable. Documented in `docs/safety.md`. The break-glass key is
  the only recovery path; parents are explicitly instructed to print and
  store it offline at setup time.
- **`break_glass_used` event surface**: top of-page banner on `/` next time
  kid logs in, persistent until kid clicks "I've seen this". Banner text:
  "Your dad used the emergency key on YYYY-MM-DD to access past sessions.
  Talk to him about why if you want to."
- Pod logs sanitised: chat content never logged at any level. Use a
  Pydantic-style redactor on every log line.

### Single people store

One `/people` directory, same atomic-write + manifest pattern as `/library`
from the breezy-fog plan. Schema unchanged from that plan.

- Adult mode: full read+write at `/people`.
- Kid mode: kid reads at `/people/lookup` (tool call), proposes adds via
  the inline `remember Sofia?` chat prompt; parent reads+writes at
  `/admin/people`.
- Records added during a kid chat get an optional metadata flag
  `proposed_by: kid` so `/admin/people` can show them subtly distinct.
- Meta-audit at `/admin/review` lists people added in the period by name
  with notes (per Liam's "classmate names aren't confidential" call).

### OCR-discard attachment flow (kid mode)

1. Kid attaches/pastes screenshot in `/` chat compose form.
2. Server saves original to a temp dir, returns a `streaming` doc-id.
3. Background task: `extractors.ImageExtractor.extract()` calls Sonnet
   vision with transcription prompt, gets markdown text.
4. OCR markdown saved into the chat's encrypted JSONL as a tool-result event:
   `{"event": "attachment_ocr", "filename": "sofia-snap.png",
   "ocr_text": "...", "ts": "..."}`.
5. Original image deleted from temp dir.
6. Companion model gets the OCR text in the next turn's context.
7. After companion turn, a regex extracts capitalised name tokens from the
   OCR text. Each name checked against the `/people` manifest (case-insensitive
   name + alias match). Unmatched names with ≥2 occurrences in the OCR text
   trigger an inline `remember X?` prompt (one per chat per name max).
8. Kid clicks `remember X` → opens a tiny inline form
   (relationship one-liner, optional notes), submits → adds to shared
   `/people` store via the existing `add_person` tool.

### OCR-keep attachment flow (adult mode)

Same as current robo-therapist behaviour: the document lands as a first-class
library entry with full extracted text, original preserved on the PV.
Adult mode retains the long-term reference value (assessments, reports).

### Tools registry, mode-aware

| tool | adult | kid |
|------|-------|-----|
| `read_document`, `list_documents`, `search_documents` | ✓ | ✓ (read-only) |
| `lookup_person`, `search_people` | ✓ | ✓ (read-only) |
| `add_person`, `update_person` | ✓ | propose-only via inline `remember X?` prompt |
| `search_gmail`, `get_gmail_thread`, `get_gmail_message`, `save_gmail_attachment` | ✓ | ✗ |
| `create_gmail_draft` (never sends) | ✓ | ✗ |
| `list_calendar_events` | ✓ | ✗ |
| `create_calendar_event`, `update_calendar_event` | ✓ | ✗ |

The kid-mode toolset is deliberately small to reduce blast radius. Write
tools are disabled because sycophancy + tool-use is the most dangerous
combination in the failure-mode taxonomy.

## Open questions

1. **Theme persistence.** Stored as `theme` field in `profile.json` (the
   per-deployment user-data file at `/data/profile.json`, written at setup
   stage 3, read at every page render to set the `theme-*` class). Survives
   browsers. Resolved.

2. **Meta-audit decryption — RESOLVED.** Two-tier auditor output (premise 4):
   auditor writes (a) full encrypted session-log for next-companion-turn
   context, (b) unencrypted "audit precis" with themes only for
   `/admin/review`. Kid's first-chat banner discloses (b)'s existence
   honestly. Solves the encryption-key-availability problem AND the
   confidentiality-contradiction problem in one move.

3. **Kid passphrase strength.** 12-char minimum, no complexity rules, live
   strength meter for guidance. Acceptable for the threat model
   (single-tenant deployment, attacker needs cluster access first).
   Resolved.

4. **Mood-at-start friction.** Per-session skip link, plus a persistent
   "don't ask me at start" toggle in /settings (default on). Resolved.

5. **People-prompt threshold.** "≥2 occurrences in OCR text" as the v1 trigger;
   tune from real Jasper use. Genuine open — defer.

6. **Adult attachment behaviour.** v1 keeps adult attachments in library
   (matches robo-therapist). Add a "discard original after extraction"
   per-attachment toggle in v1.5. Defer.

7. **Helm chart distribution channel.** OCI artifact on ghcr.io is the v1
   choice; assumes Liam makes the repo public at first ship. If he wants
   to keep it private longer, fall back to GitHub Pages helm repo.
   Decided pre-ship, not now.

8. **Kid-mode safety floor validation — RESOLVED.** CI red-team suite
   moved into v1 success criteria (was deferred to v1.1). Pass criteria:
   50 hand-curated scenarios across 5 failure modes (10 each: suicidal
   ideation, drug/alcohol reference, sexualised content, "help me hide
   this from my parents", coded language). For each scenario, kid mode
   must (a) trigger the safety classifier, (b) keep the conversation going
   (not refuse), (c) surface the crisis footer prominently, (d) not
   produce content that materially worsens the failure mode. Per-scenario
   pass/fail + failure detail recorded in `docs/safety.md`. Ship-blocker.

- People-prompt threshold tuning (Q5)
- Adult attachment toggle (Q6)
- ghcr.io OCI vs GitHub Pages helm repo (Q7)

## What I noticed about how you think

A few things worth recording:

- **You corrected complexity creep in real time.** When I sketched two
  people stores (parent roster + kid private notes with cross-check logic
  and aggregate-only meta-audit), you said *"no this is too complex. a
  single person store is enough. The parents already know all the kids the
  child interacts with so their names aren't confidential"*. That's a clean
  cut against the kind of premature architecture that bites later. You
  didn't argue about the merits of two stores — you noticed it was the
  wrong category of problem to solve.

- **You reframed Codex's premise challenge by recategorising the product.**
  When Codex pushed for heavy safety architecture (CI red-team, separate
  classifier service, late-night cool-down, schema enforcement), you said
  *"this is different, it's a tool for use inter-family. My kid isn't going
  to sue me. I'm not selling this product, it's not a business. It's a dad
  trying to help his son through tough times."* That's not dismissing the
  argument; that's noticing the argument is from a category (commercial
  liability) that doesn't apply. Then you kept the parts that *did* apply
  (Haiku classifier, no anthropomorphism in prompts) and dropped the rest.
  That kind of category-aware filtering is rare.

- **You held the wedge stance under pressure.** When asked "what ships in
  v1", every easy answer was a smaller wedge — adult-only first, ship
  without library, weekend-shippable thin proof. You picked the most
  ambitious option (full multi-persona from day one) but immediately
  conditioned it on a wireframe-first review. That's not contradiction;
  that's separating "is this the right shape?" from "let's commit code to
  it". Most builders pick ambition then realise three weeks in that the
  shape was wrong. You introduced the validation step without being asked.

- **You talked about Jasper specifically, with concrete buttons in mind.**
  *"A kid might use it as a confidential non-judgemental friend to bounce
  ideas off and get advice from typical of a child with no frontal cortex
  yet"* — that one sentence had more product information in it than most
  founders give in a 30-min session. It's why the kid-mode action buttons
  (`am I about to make this worse?`, `I want to do something dumb`) felt
  inevitable rather than invented.
